3
Is PHI encrypted in transit (TLS) and at rest?
✓ Yes — Fully
Ready to cite verbatim
"Yes — all traffic is over TLS 1.2 or higher, and the database and backups are encrypted at rest with AES-256."
  • ✓ In transit: TLS 1.2+ enforced at Vercel's edge layer on every domain. Plain HTTP not permitted — all connections auto-upgrade to HTTPS.
  • ✓ At rest: AES-256 encryption at the AWS EBS storage layer where the Supabase PostgreSQL database runs. Standard AWS default — not a custom configuration.
  • ✓ In transit (DB connection): Application-to-database connections are also SSL-encrypted. No unencrypted database access permitted.
This is a clean yes with verifiable, industry-standard configurations. Not a vague claim — both TLS and AES-256 are traceable to the underlying infrastructure providers (Vercel and AWS respectively).

5
Who has access to our members' data, and is access role-based and logged?
✓ Yes — Role-Based at Three Independent Layers

Access is enforced at three separate, independent layers. A failure or bypass at any one layer does not expose data — the next layer still enforces access control.

  • 1 Edge middleware — every HTTP request is checked at the network level before the application loads. Wrong role for this portal → immediate redirect, no data accessed.
  • 2 Application layer — every API endpoint and UI module checks the authenticated user's role before executing any query.
  • 3 Database layer (RLS) — Supabase Row Level Security policies enforce per-role data access at the database query level. Even if an API were bypassed, the DB returns only what that role is permitted to see.

6
Do you keep audit logs (who accessed what, and when)?
✓ Yes — Immutable, Append-Only, Every Mutation

Every create, update, soft-delete, and status change in the system writes an immutable audit log entry. Audit logs cannot be edited or deleted by anyone — including Super Admin.

Each entry records:

  • ✓ Who — User ID, actor type (HUMAN_USER, AI_AGENT, or SYSTEM)
  • ✓ What action — specific named action (e.g. CREATE_CASE, UPDATE_PATIENT, SIGN_CONSENT)
  • ✓ Which record — entity type and record ID
  • ✓ Before & after state — full JSON snapshot of the record before and after the change
  • ✓ IP address — originating IP of every request
  • ✓ User agent — browser and device information
  • ✓ UTC timestamp — immutable, indexed for fast lookup
This audit trail provides a complete forensic history sufficient to determine the scope, actor, and timeline of any access event — including potential breach investigation.
7
Have you experienced a breach, and what is your breach-notification process?
⚠ No Breach — Written Notification Policy Pending

Breaches to date: Zero. No PHI is in the system (Phase 0 by design), which eliminates breach risk at the data layer. No unauthorized access incidents have been recorded.

Breach notification process: A formal written policy does not yet exist as a standalone document. What is in place today:

  • ✓ Sentry real-time error monitoring — alerts the engineering team immediately on production anomalies
  • ✓ Complete immutable audit log — full forensic trail to establish scope and timeline of any incident
  • ✓ Role isolation — a compromised account cannot access data outside its role scope
  • ! Written Incident Response Policy — not yet documented
  • ! HIPAA 4-factor breach risk assessment process — not yet documented
  • ! Individual patient notification procedure (60-day requirement) — not yet documented
  • ! HHS reporting procedure — not yet documented
HIPAA Breach Notification Rule (45 CFR §164.400–414): Requires notification to affected individuals within 60 days of discovery. HHS must be notified immediately if 500+ individuals are affected, or via an annual log if fewer than 500. These written procedures are a deliverable of the HIPAA consultant engagement scheduled for Phase 1.
