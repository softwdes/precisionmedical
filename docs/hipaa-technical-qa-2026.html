LienMaster V2 — Phase 0
This document provides precise, technical answers to four core HIPAA security questions as they apply to the LienMaster V2 platform built for Precision Medical. All answers reflect the current state of the system as of June 2026. The platform is currently in Phase 0 — no real Protected Health Information (PHI) has been introduced into any production system. Every infrastructure and access control safeguard described below is live and operational, awaiting the formal Business Associate Agreement (BAA) signing process before PHI is activated.
1
Is member PHI encrypted in transit and at rest?
Direct answer — ready to cite
"Yes — all traffic is over TLS 1.2 or higher, and the database and backups are encrypted at rest with AES-256."
What makes this answer credible: TLS is enforced at the Vercel edge layer on every domain — no plain HTTP is permitted. AES-256 at rest is provided by AWS at the EBS storage layer, where the Supabase PostgreSQL database runs. Both are verifiable, standard industry configurations — not custom claims.
In Transit
✓ Yes — Fully Encrypted

All traffic across every portal — admin.lienmaster.net, pmtc.lienmaster.net, clinic.lienmaster.net, and all future portals — is served exclusively over HTTPS with TLS encryption enforced at the infrastructure level. Let's Encrypt SSL certificates are active on all domains. Plain HTTP connections are not permitted on any endpoint and are automatically redirected to HTTPS. This includes all API calls, authentication tokens, and any data exchanged between the browser and the server.

At Rest
✓ Yes — AES-256 at Storage Layer

The database runs on Supabase, hosted on Amazon Web Services (AWS) in the us-east-2 region (Ohio). AWS encrypts all data at rest using AES-256 by default at the EBS (Elastic Block Store) storage layer. This applies to every table in the system including patient records, clinical notes, case data, and billing information. Application-level data is also transmitted to the database over encrypted SSL connections.

Note on current PHI status: The encryption infrastructure is fully in place and operational. However, no real PHI exists in the system today (Phase 0). The encryption protections will be active and enforced at the moment real patient data is introduced in Phase 1.

3
Is access role-based and logged (audit trail)?
Role-Based Access Control (RBAC)
✓ Yes — 3 Independent Layers

Access control is enforced at three separate, independent layers. A user must pass all three to access any data. This defense-in-depth approach means that a failure or bypass at any single layer does not expose data.

1

Middleware Layer — Portal Access

Every HTTP request is intercepted at the network edge before it reaches the application. Users without the correct role for a given portal are immediately redirected or blocked — no data is loaded, no queries are made. A billing staff member attempting to access the doctor portal is stopped here.

2

Application Layer — Feature Access

Inside each portal, every module, API endpoint, and action checks the authenticated user's role before executing. A billing coordinator (CONTADOR role) cannot access clinical notes. A provider (PROVIDER role) cannot view payroll data. A lawyer (LAWYER role) can only see their own firm's cases.

3

Database Layer — Row Level Security (RLS)

Supabase Row Level Security policies are active at the database level. Even if an API were somehow bypassed, the database itself enforces per-role data access rules at the query level. A query from a LAWYER account will only return rows belonging to that lawyer's firm — the database enforces this regardless of how the request was made.

The platform has 7 distinct roles: SUPER_ADMIN, ADMIN, CONTADOR (Accounting), EMPLOYEE, PROVIDER (Doctor / MA), LAWYER, and AUDITOR_AI. Each role has a precisely defined scope of access. No role has blanket access to the entire system except SUPER_ADMIN, whose actions are themselves fully logged.

Audit Trail
✓ Yes — Every Mutation Logged

Every create, update, soft-delete, and status change writes an immutable, append-only audit log entry. The audit log cannot be edited or deleted by anyone in the system, including SUPER_ADMIN.

Each audit log record contains:

  • ✓ Who — user ID and actor type (HUMAN_USER, AI_AGENT, or SYSTEM)
  • ✓ What action — specific action name (e.g. UPDATE_CASE, CREATE_PATIENT, SIGN_CONSENT)
  • ✓ What entity — entity type and record ID that was affected
  • ✓ Before & after state — full JSON snapshot of the record before and after the change
  • ✓ IP address — originating IP of the request
  • ✓ User agent — browser/device that made the request
  • ✓ Timestamp — UTC timestamp, immutable
This audit trail provides a complete forensic history of every data change in the system — sufficient to determine the scope, actor, and timeline of any potential breach or unauthorized access event.
4
Do you have a breach-notification process?
Honest Assessment
⚠ Partially — In Progress

A formal, written Breach Notification Policy does not yet exist as a standalone document. This is a known gap that is on the Phase 1 pre-PHI checklist. Below is what exists today and what needs to be completed.

What Is Already in Place
  • ✓ Sentry error monitoring — alerts the engineering team in real time when production errors occur. Any anomalous access pattern or system error triggers an immediate notification. (Note: Sentry BAA must be signed before it can process PHI.)
  • ✓ Complete audit log — every data access and change is recorded, providing a full forensic trail to establish the scope, timing, and actor of any potential incident.
  • ✓ Role isolation — the access control architecture limits the blast radius of any single compromised account. A compromised billing account cannot access clinical records.
  • ✓ No PHI currently in system — there is no real patient data to breach at this time. Phase 0 by design eliminates breach risk at the data layer.
What Needs to Be Established Before PHI Activation
  • ! Written Incident Response Policy — naming the responsible person, escalation steps, and internal response timeline
  • ! Breach risk assessment process — the HIPAA 4-factor test to determine whether an incident qualifies as a reportable breach
  • ! Individual patient notification procedure — written process for notifying affected patients within 60 days of discovery
  • ! HHS reporting procedure — immediate notification if 500+ individuals affected; annual log submission if under 500
  • ! Business Associate notification obligations — written process to notify Supabase, Vercel, and Twilio if a breach involves their systems
  • ! Staff training records — documentation that all staff with PHI access have completed HIPAA training
HIPAA Breach Notification Rule (45 CFR §164.400–414) requires: Notification to affected individuals within 60 days of breach discovery. Notification to HHS (Department of Health & Human Services) — immediately if 500+ individuals are affected, or as an annual log submission if fewer than 500. These requirements are standard across all covered entities and are addressed through the HIPAA consultant engagement already listed as a Phase 1 blocking gate.

🔒 Executive Summary — Where We Stand

Technically, the platform is architecturally ready for PHI. Encryption in transit and at rest is fully operational. Role-based access control is enforced at three independent layers. An immutable audit trail records every data change. These are not future plans — they are live and working today.

The remaining gaps are administrative and legal, not technical. Before any real patient data enters the system, three things must happen: (1) sign BAAs with Supabase, Vercel, and Sentry, (2) produce a written Breach Notification Policy, and (3) complete a HIPAA consultant review. These are standard procurement and legal steps required of every healthcare software platform — they are already on our Phase 1 gate checklist.

The system was built with this moment in mind. No shortcuts were taken in the architecture. The decision to start with zero PHI (Phase 0) was deliberate — it allows the team to build and stabilize the platform before introducing protected data, which is the correct order of operations from both a technical and compliance standpoint.
